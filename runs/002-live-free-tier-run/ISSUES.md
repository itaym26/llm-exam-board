# Run 002: Issues Log

Each entry: what caused the problem, how it was resolved, how we proceeded afterward. Logged as they happened, in order.

## Issue 1: `gemini-2.0-flash` returns 429 RESOURCE_EXHAUSTED on a fresh free-tier key

**Cause:** Before running the pipeline, the Gemini API key was smoke-tested directly (a single `generate_content` call, bypassing the pipeline). The default model configured in `gemini_client.py`, `gemini-2.0-flash`, returned `429 RESOURCE_EXHAUSTED` ("You exceeded your current quota") on the very first call ever made with this key — this is not a transient rate limit, since a brand-new key cannot have used up a quota already. `gemini-2.0-flash-lite` failed the same way. This indicates Google no longer grants free-tier quota to the `gemini-2.0-*` model family for new keys (likely deprecated/phased out in favor of the 2.5 series since these client wrappers were written).

**How we solved it:** Tested `gemini-2.5-flash` and `gemini-1.5-flash` directly. `gemini-2.5-flash` succeeded immediately ("SUCCESS -> OK"). `gemini-1.5-flash` failed differently — `404 NOT_FOUND` ("is not found for API version v1beta") — confirming that model has been fully retired, not just quota-limited.

**How we proceeded:** Changed the default `model` parameter in `examples/gemini_client.py`'s `make_gemini_client` from `"gemini-2.0-flash"` to `"gemini-2.5-flash"`, and updated the hardcoded model string in `examples/run_free_demo.py`'s judge instantiation to match. This was committed to the repository (not just patched locally) so future runs don't repeat this discovery. No change was needed on the Groq side — `llama-3.1-8b-instant` authenticated and responded correctly on the first attempt.

## Issue 2: `QuestionGenerator: Teacher LLM returned unparsable output` on the first real run

**Cause:** The first actual execution of `examples/run_free_demo.py` (with both keys fixed per Issue 1) failed with `RuntimeError: QuestionGenerator: Teacher LLM returned unparsable output for topic 'Multithreading'`, raised from `generators.py`'s `generate_test_case`. To diagnose, the Teacher LLM was called directly (bypassing `ResilienceUtils`) and its raw response was saved to `teacher_raw_debug.txt` for inspection. The response was a complete, well-formed-looking JSON object wrapped in a markdown fence — but `json.loads` failed with `Invalid control character at: line 3 column 24`. The real Groq-hosted Llama model had embedded its multi-line Python `golden_reference` code as a JSON string value using **literal, unescaped newline characters** instead of the `\n` escape sequence strict JSON requires. This is a genuinely common LLM output pattern (not unique to this one call) whenever a JSON string field is asked to hold multi-line code, and `ResilienceUtils.clean_json_response` at the time had no repair strategy for it — its three existing strategies (direct parse, balanced-bracket extraction, trailing-comma repair) all assume the string *contents* are already valid.

**How we solved it:** This was treated as a real gap in the shared `ResilienceUtils` module, not a one-off workaround for this run. Added a fourth recovery strategy, `__escape_control_chars_in_strings`, to `src/llm_exam_board/resilience.py`: a string-literal-aware scan (structured like the existing balanced-bracket scanner, tracking `in_string`/`escape_next` state) that rewrites raw `\n`, `\r`, and `\t` characters into their escaped form *only* when they occur inside an unescaped string literal, leaving JSON structure elsewhere untouched. Verified the fix directly against the saved `teacher_raw_debug.txt`: parsing now succeeds and recovers the full `question`, a 35-line `golden_reference`, and 4 rubric items. Re-ran `examples/run_demo.py` (the mocked run 001 suite) afterward to confirm no regression — all three scenarios still pass.

**How we proceeded:** Committed the `resilience.py` fix to the repository before re-attempting the live run, so this fix benefits every future run (and any real usage of `JudgeEngine`/`QuestionGenerator`), not just this one script. Then re-ran `examples/run_free_demo.py` from scratch.

## Issue 3: The same `RuntimeError` reappeared immediately after the Issue 2 fix

**Cause:** Re-running `examples/run_free_demo.py` right after committing the Issue 2 fix failed again with the identical `RuntimeError: QuestionGenerator: Teacher LLM returned unparsable output`. This was concerning, since it suggested the fix might not actually be effective. Diagnosis: a fresh, direct call to the Teacher LLM was made and fed straight through `ResilienceUtils.clean_json_response` — it parsed successfully. Called again 8 times in a row directly — all 8 parsed successfully. The one variable that changed between the failing run and these successful diagnostic calls was that `__pycache__/` directories (containing compiled `.pyc` bytecode from before the `resilience.py` edit) were cleared immediately before the successful diagnostic batch.

**How we solved it:** Concluded the most likely explanation is that the failing re-run picked up stale compiled bytecode for `resilience.py` predating the Issue 2 fix (Python normally invalidates `.pyc` files via source `mtime`/hash checks, but this cannot be fully ruled out as the cause here given several rapid edits to the same file in quick succession). Ran `find . -iname "__pycache__" -exec rm -rf {} +` to remove all compiled bytecode across the project, then repeated the 8-call stress test above, which passed 8/8.

**How we proceeded:** Adopted a rule for the remainder of this run (and worth keeping for future runs): clear `__pycache__` before any run that follows a source-code change, to eliminate stale bytecode as a confound when diagnosing a failure. Re-ran `examples/run_free_demo.py` once more with a clean cache.

## Issue 4: The Truth Gate zeroed a functionally correct student answer

**Cause:** This run completed successfully (exit code 0), but inspecting `output.json` shows the Truth Gate triggered on *both* judges, zeroing a 100-point raw score down to 0. The student's answer (real, from `llama-3.1-8b-instant`) is a genuinely correct, idiomatic thread-safe `Counter` class — it acquires `self._lock = threading.Lock()` and uses `with self._lock:` in both `increment` and `get_value`. However, the Teacher (`llama-3.3-70b-versatile`) authored rubric `validation_pattern`s that hardcode the exact identifier name `mutex` (e.g. `"self\\.mutex\\s*=\\s*threading\\.Lock\\(\\)"`, `"with\\s+self\\.mutex:"`). The student used `self._lock` instead of `self.mutex` — semantically identical, but textually different — so every critical pattern legitimately failed to match, and the Truth Gate correctly did exactly what it was built to do (statically enforce its given patterns) against a rubric that was too rigidly worded.

**How we solved it:** This is not a software bug — `JudgeEngine`'s Truth Gate behaved exactly as designed given the patterns it was handed. The root cause sits one layer up, in how `QuestionGenerator` prompts the Teacher to author `validation_pattern` values: nothing in `generators.py`'s prompt (see `__build_generation_prompt`) instructs the Teacher to write patterns that are robust to naming variation (checking for `threading\.Lock\(\)` instantiation and a `with\s+\w+\.\w+:` guard in general, rather than binding to one specific variable name). We did not patch this during the run, to avoid scope creep on what was meant to be a small, contained run, and because a prompt change like this deserves its own validation pass rather than a rushed edit.

**How we proceeded:** Documenting this as a known limitation rather than silently accepting the 0.0 consensus score at face value. Recommended follow-up (not yet implemented): update `QuestionGenerator.__build_generation_prompt` in `generators.py` to explicitly instruct the Teacher to write `validation_pattern` regexes that check for the *pattern of use* (e.g., any lock attribute name, guarded by `with`) rather than one specific identifier — then re-run to confirm the false trigger no longer occurs on equivalent-but-differently-named implementations.

## Issue 5: The Gemini judge's response could not be parsed

**Cause:** `output.json` shows `judge-gemini-flash`'s `feedback_summary` is the generic fallback string `"Judge LLM response could not be parsed."`, meaning `ResilienceUtils.clean_json_response` returned `None` for whatever Gemini actually returned for the grading prompt. Because `JudgeEngine.grade` was not modified to persist the raw pre-parse response anywhere, the actual raw text is gone — it cannot be inspected retroactively, and re-calling the API now would not reproduce the same (stochastic) output.

**How we solved it:** We did not attempt a fix mid-run, since we have no artifact to diagnose against. We did notice a related, more important design point while investigating: when `clean_json_response` returns `None`, `JudgeEngine.grade` falls back to treating the judge as having proposed *zero deductions* (see `judge_engine.py`, `llm_deductions = []`), which computes a `raw_llm_score` of full marks (100.0 here). In this specific run that default was harmless, because the Truth Gate independently zeroed the score anyway — but in a run where the Truth Gate does *not* trigger, an unparsable judge response would silently contribute a perfect score to the ensemble, indistinguishable from a judge that genuinely reviewed the answer and found nothing wrong.

**How we proceeded:** Documenting both the missing raw-response capture and the "unparsable defaults to a perfect score" behavior as follow-up items, rather than patching them under time pressure at the end of this run:
1. Add optional raw-response logging to `JudgeEngine` (or capture it at the call site, as `run_free_demo.py` already does for other intermediate artifacts) so a future parse failure can actually be diagnosed.
2. Consider whether an unparsable judge response should instead be treated as an explicit "abstention" (excluded from the ensemble, or flagged, rather than silently scored as if nothing was wrong) — a genuine design question worth a deliberate decision, not a quick patch.
