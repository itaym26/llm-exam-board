# Run 003: Issues Log

Each entry: what caused the problem, how it was resolved, how we proceeded afterward. Logged as they happened, in order. This run surfaced four distinct real problems -- more than run 002 -- precisely because it exercised 10 independent real generations instead of 1, and used Gemini as a Teacher for the first time (previously only as a Judge).

## Issue 1: The Truth Gate fired on all 4 successful Groq-taught test cases, despite the run 002 naming fix

**Cause:** Inspecting `output.json`, every one of the 4 test cases that completed showed `truth_gate_triggered=true` on both judges. Closer inspection of the rubric `validation_pattern`s revealed the naming-agnostic fix from run 002 (commit `e7d5aaa`) *was* being followed some of the time (e.g. `"self\\.\\w+\\s*=\\s*threading\\.Lock\\(\\)"` correctly matched the student's `self._lock` in one test case, and did not falsely trigger) -- but several *other* critical patterns in the same rubrics were simply malformed regardless of naming, most commonly missing the `self` parameter in a method-signature pattern (e.g. `"def __init__\\(\\):\\s*..."` instead of `"def __init__\\(self\\):\\s*..."`), which cannot match *any* valid Python method, including the Teacher's own golden reference.

**How we solved it:** This is documented as a finding, not something we patched by chasing this one exact pattern -- because the deeper, systemic causes turned out to be Issue 2 and Issue 4 below (patterns not accounting for multi-line code, and rigid whole-signature/whole-call matching in general). Fixing those two addresses this category of failure more broadly than special-casing "missing `self`" would have.

**How we proceeded:** Folded into the broader Teacher-prompt strengthening (Issue 4) and the new Audit Gate self-check (Issue 3), rather than treated as a standalone fix.

## Issue 2: `JudgeEngine.grade` crashed entirely with `unbalanced parenthesis at position 27`

**Cause:** One of the 5 Groq-taught test cases (index 3) failed with a raw `re.error` propagating out of `JudgeEngine.grade`, aborting that whole test case (caught only by this script's own outer try/except, not by the library). The Teacher had generated a `validation_pattern` that was not valid regex syntax at all -- `__apply_truth_gate` called `re.search(item.validation_pattern, answer_text)` directly, with no protection against the Teacher producing a syntactically invalid pattern.

**How we solved it:** Added `JudgeEngine.check_pattern` (`src/llm_exam_board/judge_engine.py`), a static method wrapping `re.search` in a `try/except re.error`, returning `None` (rather than raising) when the pattern itself is invalid. `__apply_truth_gate` now treats `None` as "inconclusive" -- it does *not* trigger an override for a pattern that cannot even be evaluated, since penalizing the student for a Teacher-authored defect would be exactly the kind of false positive the Truth Gate is supposed to prevent, not cause.

**How we proceeded:** Committed the fix to the library (not just this script). `check_pattern` was also written as a public, reusable static method specifically so the Audit Gate (Issue 3) could share the same safe logic rather than duplicating a second unguarded `re.search`.

## Issue 3: Broken patterns weren't caught before reaching the Student -- extended the Audit Gate

**Cause:** Several critical `validation_pattern`s in this run's rubrics could not have matched *even the Teacher's own golden_reference* -- not just the Student's answer. The Audit Gate's stated purpose ("ensures tasks generated by the Teacher are solvable") had never actually been checking this; it only verified a non-empty golden reference and a non-empty rubric existed at all, not that the rubric was internally usable.

**How we solved it:** Extended `EvaluationOrchestrator`'s private `__run_audit_gate` (`src/llm_exam_board/orchestrator.py`) to call the new `JudgeEngine.check_pattern` against `test_case.golden_reference` for every critical rubric item, rejecting the TestCase outright if any critical pattern is invalid regex or simply does not match the reference solution. This is a genuine strengthening of what "solvable" means, directly using the finding from Issue 1.

**How we proceeded:** Since a real Teacher will occasionally fail this stricter self-consistency check by chance (verified directly -- see Issue 4's diagnostics below), also added a bounded retry: `EvaluationOrchestrator.run_single_evaluation` now asks the Teacher to regenerate up to `__MAX_AUDIT_GATE_ATTEMPTS = 3` times before raising, instead of failing on the first rejection. The three example scripts' locally-duplicated audit checks were updated to match, and `run_two_teachers_demo.py` additionally gained the same bounded-retry wrapper (`generate_audited_test_case`).

## Issue 4: Even naming-agnostic patterns kept failing self-consistency -- root cause was `.` not matching newlines

**Cause:** After implementing the Audit Gate self-check (Issue 3) and testing it against fresh real Groq generations, 0 out of 3 test cases passed self-consistency -- worse than expected. Direct inspection showed patterns like `"def \\w+\\(self, \\w+\\):.*\\.put\\(\\w+\\)"` failing to match code where the method signature and the `.put(...)` call were on different lines (the overwhelmingly common case for any real method body). Verified directly: `re.search(r"def \w+\(self, \w+\):.*\.put\(\w+\)", "def add_task(self, task):\n        self.queue.put(task)")` returns `None` without `re.DOTALL`, and a match with it -- because Python's `.` does not match `\n` by default. This is a systemic issue, not a one-off: nearly every multi-line pattern the Teacher writes (which is most of them, since checking "is X called somewhere in this method" naturally spans lines) was silently failing regardless of the naming or content it was checking.

**How we solved it:** Changed `JudgeEngine.check_pattern` to call `re.search(pattern, text, re.DOTALL)` instead of `re.search(pattern, text)`. Re-tested against 5 fresh real Groq generations: self-consistency failures dropped from 3-4 broken patterns per test case down to 1-2, and the remaining failures were a *different*, narrower class of problem (patterns using `\w+` where the real code used a dotted attribute access like `self.foo`, or a hardcoded numeric literal like `\d+` where the code used a named parameter instead) -- confirmed these are inherent, irreducible LLM authoring inconsistency, not a fixable systemic bug, which is exactly why the Issue 3 retry loop exists: some fraction of generations will always need a second attempt.

**How we proceeded:** Also strengthened the Teacher prompt itself (`generators.py`) as defense in depth: explicitly extended the "don't hardcode a name" rule to method/class names in addition to variables, required the question text to state any name a pattern depends on, instructed the Teacher to prefer short distinctive fragments over rigid whole-signature/whole-call patterns, and added an explicit self-check instruction ("mentally verify every critical pattern against your own golden_reference before responding"). This won't reach 100% -- Issue 3's retry loop is the actual safety net -- but it should reduce the retry rate.
